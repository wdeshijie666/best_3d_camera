# 3D 相机底层驱动与上层 SDK 架构设想

## 1. 目的与实现原则

- **目标**：构建 **3D 相机底层驱动** 与 **上层 SDK**，完成投采控制、与检测/重建流水线协同、向应用层提供统一接口与可视化能力。
- **实现原则**：
  - **整体以 C++ 为主**：中枢、相机驱动封装、检测/重建服务、上层 C++ SDK、Viewer 均在 C++ 技术栈内实现，避免脚本语言带来的运行时依赖与分发复杂度。
  - **跨平台优先**：接口与 IPC 设计按 **Windows / Linux**（及必要时 macOS）可移植考虑；**首发落地平台可为 Windows**，但实现上通过抽象层隔离 OS 专用 API（共享内存命名、服务拉起、权限模型等）。
  - **Python 暂不纳入**：当前阶段不提供 Python 绑定、不以 Python 做编排或 Viewer；若未来需要，可在 C++ SDK 稳定后再单独评估 **pybind11** 等封装，本文档不展开。
  - **公共基础设施统一**：**中枢、相机驱动服务、检测、重建、Viewer、SDK 宿主进程** 均链接同一套可复用模块（建议独立 **静态/动态库**），包含 **统一日志** 与 **统一异常与崩溃捕获**（见 **§4.6**、**§5.7**），禁止各可执行文件各自实现一套互不兼容的日志格式与崩溃落盘逻辑。

## 2. 架构分层（原文档整理）

### 2.1 底层服务（中枢）

- **职责**：相机驱动层 + 与上层 SDK 的交互服务（对外 RPC 建议 **gRPC**，与语言无关、便于多客户端集成）。
- **定位**：底层中枢，负责指令收发与数据传输；管理相机投采数据；与 **检测服务**、**重建服务** 通过 **共享内存** 交换大块数据。
- **单次采集流程**：接收上层「单次采集」指令后，调度相机驱动以及检测、重建进程，将 **原始数据**、**中间结果**、**最终结果** 汇总并回传给上层 SDK。

### 2.2 相机驱动

- **硬件**：光源 + 工业相机（如暂定 **大恒**），通过 **串口** 对一体机下发投采指令。
- **部署**：初代可部署在一体机上；可做成 **带界面的本地服务**，便于现场调试（参数、触发、日志）。
- **功能**：配置相机参数（曝光、增益、ROI、触发模式等）；向一体机发串口指令完成投采；将帧/点云等交给中枢做后续分发。

### 2.3 数据检测服务

- **形态**：独立于底层服务的进程。
- **通信**：与底层服务通过 **共享内存** 交互；接收原始数据，输出检测结果回传中服务。
- **职责**：缺陷/几何/标定相关等 **检测** 算法（具体算法与 SLA 由产品定义）。

### 2.4 数据重建服务

- **形态**：独立于底层服务的进程。
- **通信**：共享内存与底层服务交互；接收原始数据，输出重建结果回传。
- **职责**：深度/点云/网格等 **重建** 管线。

### 2.5 上层 SDK

- **能力**：设备操作、调参、结果订阅/拉取。
- **版本**：区分 **开发者版** 与 **用户版**——开发者接口权限更高，可访问更多数据类型与诊断信息（原始缓冲、中间 tensor、详细标定参数等）。

### 2.6 相机 Viewer

- **定位**：基于上层 SDK 的参考实现 / 标配工具，人机交互与可视化。
- **能力**：与 SDK 接口对齐的操作与调参；**深度图**、**重建结果**、**模型 + 缺陷** 展示；数据保存；常用后处理入口。
- **技术倾向**：**Qt（C++，建议 Qt 6）+ OpenCV + VTK + SQLite**；与上层 SDK 同一进程或同库链接，避免跨语言边界。

## 3. 可行性评估

| 维度 | 结论 | 说明 |
|------|------|------|
| **总体** | **可行** | 多进程 + 共享内存传大图/点云 + gRPC 传控制与小 payload，在工业视觉与 3D 方案中常见；C++ 全栈在 Windows/Linux 上均成熟。 |
| **跨平台** | **可行，需抽象层** | 共享内存与同步：**POSIX shm**（Linux 等）与 **文件映射**（Windows）语义不同，可用 **Boost.Interprocess**、**Qt QSharedMemory**（见 §4.1.1）或自研薄封装；相机与串口仍可能因厂商 SDK 存在 **按 OS 分包** 的插件式实现。 |
| **吞吐与延迟** | **需专项设计** | 高分辨率多帧、多路结果若全经 gRPC 传体会成为瓶颈；大块数据应固定在 **共享内存 / 内存映射文件**，RPC 只传 **句柄、偏移、序号、元数据**。 |
| **稳定性** | **可行，依赖进程隔离** | 检测/重建崩溃不应拖垮采集中枢；需 **看门狗、重启策略、会话状态机**。 |
| **大恒相机** | **可行** | 使用官方 **Galaxy SDK（GxIAPI）**；注意 **运行时驱动、GenICam/GigE/USB3** 与目标机型一致。 |
| **串口 + 一体机** | **需注意可靠性** | 串口丢包、乱序、超时；建议协议层 **帧头/校验/ACK**，中枢侧 **重试与超时策略**。 |
| **开发者/用户双 SDK** | **可行** | 通过 **编译期裁剪**、**单独包** 或 **运行时 License/能力令牌** 控制 API 与数据可见性，需在安全与易用间折中。 |
| **统一日志与崩溃诊断** | **可行，强烈建议** | 多进程排障依赖 **一致字段、路径与崩溃包格式**；单独库 + `main` 入口统一初始化可降低现场支持成本（见 **§4.6**、**§5.7**）。 |

**结论**：架构方向合理；工程风险主要集中在 **共享内存协议设计**、**背压与生命周期**、**大数据路径零拷贝** 以及与 **厂商 SDK/驱动** 的版本锁定。

## 4. 建议技术栈（补充）

### 4.1 底层服务（中枢）与进程间通信

| 类别 | 建议选项 | 备注 |
|------|----------|------|
| 实现语言 | **C++17        
法进程协同均在 C++ 内完成；可选 **Rust** 独立模块经 C ABI 接入，非必选。 |
| RPC | **gRPC C++ + Protocol Buffers** | 流式用于 **元数据/进度/日志**；大块数据仍走共享内存，不经 RPC 本体。 |
| 共享内存（跨平台） | **Boost.Interprocess**、**Qt `QSharedMemory`**（Qt 6 Core）或自研 **OS 抽象** | 见下节 **§4.1.1** 选型对比；无论选哪种，对外仍建议统一为 **一块连续映射区 + 固定头 + 槽位环** 的协议，而非零散多段 `QSharedMemory`（除非自行管理多段映射表）。 |
| 同步原语 | **进程间 Mutex / 条件变量 / 命名信号量**（Boost 或 Qt **`QSystemSemaphore`** 等） | 环形缓冲仍推荐 **序列号 + `std::atomic` / 内存序**；若全链路已依赖 Qt Core，可与 `QSharedMemory` 同栈选型。 |
| 日志与可观测 | 见 **§4.6**（统一 **spdlog** 封装 + 可选 **ETW**） | 全模块共用同一日志库与初始化流程，避免格式分裂。 |

#### 4.1.1 使用 Qt `QSharedMemory` 是否可行

**可以。** `QSharedMemory`（`Qt6::Core`）在 **Windows / Linux / macOS** 上提供同一套 API，由 Qt 屏蔽底层差异，有利于减少自研 OS 分支代码。

**适用前提与权衡**：

| 方面 | 说明 |
|------|------|
| **依赖范围** | 检测、重建、中枢等进程若都通过共享内存与中枢对话，则需 **链接 Qt Core**（不必链接 Widgets）。算法侧仅增加 Core 依赖，体量与许可（LGPL/商业）需纳入发布策略。 |
| **与「无 Qt」算法栈** | 若希望检测/重建进程 **零 Qt 依赖**（仅 OpenCV/PCL/ONNX 等），则更合适在中枢侧用 **Boost.Interprocess** 或原生映射，避免为 IPC 单独引入 Qt。 |
| **容量与模型** | `QSharedMemory` 通常管理 **一段** 连续区域；**环形多缓冲** 仍建议在 **单段内** 用头结构 + 偏移实现（与 Boost 方案一致），而不是为每一帧创建/销毁多个 `QSharedMemory` 对象。 |
| **健壮性** | 需注意 **attach/detach 生命周期**、异常退出后的 **残留键（native key）**、Linux 上 **`/dev/shm` 与权限**；生产环境建议配合 **固定命名规范 + 崩溃清理或重启策略**（与任意共享内存方案相同）。 |
| **与 Viewer** | Viewer 已计划 **Qt**，若中枢也选 Qt Core 做 IPC，则 **调试工具与中枢可共享同一套封装类**，降低维护成本。 |

**简要结论**：**若接受「中枢 + 检测 + 重建」均依赖 Qt6::Core**，用 `QSharedMemory` 做跨平台共享内存是合理选项；**若算法进程坚持无 Qt**，则中枢与算法之间仍宜用 **Boost.Interprocess / 原生映射**，Viewer 仅通过 **gRPC 元数据 + 不直接 attach 算法段**（或单独只读小段预览）交互。

### 4.2 相机与串口

| 类别 | 建议选项 |
|------|----------|
| 大恒采集 | **Galaxy SDK（GxIAPI）** C++ API；在统一 **ICameraDevice** 接口后按 OS/机型挂接厂商库（若某平台无官方库，该后端仅编译目标平台）。 |
| 串口 | **Boost.Asio** 或 **asio**（跨平台）；Windows 亦可封装 **CreateFile** COM 口，建议仍收于同一 **ISerialPort** 抽象内。 |
| 图像内存 | **零拷贝**：厂商 SDK buffer → 直接写入共享内存槽位，避免多余 `memcpy`（在 SDK 允许的前提下） |

### 4.3 检测与重建服务

| 类别 | 建议选项 |
|------|----------|
| 传统 CV | **OpenCV**（C++），CUDA 模块按显卡与授权情况选用 |
| 点云/网格 | **PCL**（C++）；或其它 C++ 点云/网格库，按 license 与依赖体量选型 |
| 深度学习（若需要） | **ONNX Runtime C++**；NVIDIA 场景可 **TensorRT** | 与检测进程同进程或子模块，避免在中枢进程加载重模型 |
| 并行 | **OpenMP**、**TBB**、GPU stream | 与共享内存生产者-消费者模型对齐 |

### 4.4 上层 SDK

| 类别 | 建议选项 |
|------|----------|
| C++ SDK（主交付） | 与 **gRPC C++** 生成代码同仓或同发行包，**语义化版本**；对外头文件与链接库清晰分层（用户版 / 开发者版可通过 **不同导入库或宏** 裁剪符号）。 |
| 其他语言（可选、非本期） | 工厂侧若需 **.NET**，可用 **Grpc.Net.Client** 直连同一 proto；**Python 等绑定本期不纳入** 产品范围。 |
| 开发者/用户差异 | **不同 proto 包** 或 **服务端按 token 裁剪字段**；敏感原始数据仅开发者版客户端可订阅 |

### 4.5 Viewer（Qt + OpenCV + VTK + SQLite）

| 类别 | 建议选项 |
|------|----------|
| UI | **Qt 6**（**Widgets 或 QML + C++ 后端**），与上层 SDK 静态/动态链接一致，便于单安装包分发 |
| 可视化 | **VTK**（点云/网格）；**OpenCV**（深度伪彩、简单算子与小工具） |
| 数据落地 | **SQLite**（配置、标定摘要、测量历史）；大文件仍用 **文件系统 + 路径索引** |
| 性能 | 大图显示用 **GPU 纹理** 或 VTK 路径，避免 UI 线程阻塞 |

### 4.6 公共可复用模块：统一日志与异常/崩溃捕获

**目标**：所有进程（中枢、检测、重建、相机调试服务、Viewer、命令行工具等）使用 **同一套 API 与行为**，便于联调、现场抓包与自动化分析。

#### 4.6.1 统一日志模块（建议独立库，如 `platform_diag` / `vision_log`）

| 项 | 建议 |
|------|------|
| **实现基座** | **spdlog**（同步/异步 sink、轮转、多 sink）；封装为 **头文件 + 少量 .cpp** 的薄层，对外只暴露 **初始化、宏或流式 API、子模块 tag**。 |
| **字段规范** | 每条日志至少包含：**UTC/本地时间（ISO8601）**、**进程名与 PID**、**模块 tag**（如 `core`/`grab`/`detect`/`recon`/`viewer`）、**级别**、**线程 id**（可选）、**采集会话 id / trace id**（与 gRPC 或共享内存头一致更佳）。 |
| **输出** | **滚动文件**（按大小/日期）；可选 **控制台**（调试构建）；开发者版可开 **JSON 行** 便于 ELK 类工具；用户版默认 **Warn**、路径与保留天数可配置。 |
| **性能** | 热路径使用 **异步后端 + 队列上限**（满策略：丢最旧或阻塞，需可配）；避免在相机回调里同步写盘。 |
| **与 Qt** | Viewer 使用 **`qInstallMessageHandler`** 将 `qDebug`/`qWarning` 等 **转发到同一 spdlog sink**，避免双套日志文件。 |
| **敏感信息** | 配置中口令、License 片段在公共层做 **统一脱敏**（如仅打印 hash 前后缀）。 |

#### 4.6.2 统一程序异常与崩溃捕获模块（与日志库同仓或同库）

| 项 | 建议 |
|------|------|
| **C++ 未捕获异常** | 在 **`main` 外层** `try/catch(std::exception)` 记录 `what()` + 栈（见下）；对 **`std::set_terminate` 处理器** 记录 `std::current_exception` 信息。 |
| **Windows 原生崩溃** | 可自研 **`SetUnhandledExceptionFilter` + DbgHelp MiniDump**；或集成 **Breakpad / Crashpad**，由库接管并落盘。 |
| **Linux / macOS 崩溃** | **Breakpad / Crashpad** 在 Unix 侧通过 **信号处理 + 外进程/约定格式写盘** 收集崩溃；与 Windows 分支共用 **同一套上报目录与符号化流程**（见 **§4.6.3**）。若不引入第三方库，则需自管信号安全与栈回溯，维护成本更高。 |
| **初始化时机** | 在 **`main` 开头**、**任何 worker 线程与第三方 SDK 初始化之前** 安装过滤器与 terminate 处理器；与 **日志系统初始化顺序** 固定（先日志后 dump，确保 dump 路径可写）。 |
| **输出约定** | 与日志同根目录，例如 `logs/<date>/` 下 **`*.log`** + **崩溃产物**（Windows 上多为 **`.dmp`**，Linux/mac 上 Breakpad 另有约定文件名）；附 **`.txt` 摘要** 更佳；文件名带 **可执行文件名与版本**。 |
| **与多进程** | 各进程独立落盘；中枢可监控子进程 **异常退出码** 并写 **一条关联日志**（含子进程名与最后已知会话 id）。 |

**依赖注意**：崩溃模块会链接平台相关库（Windows **dbghelp**、**psapi** 等）；建议 **条件编译** 或 **单独翻译单元**，避免污染无头服务的最小依赖（仍建议最终所有服务带同一崩溃捕获，便于现场）。

#### 4.6.3 spdlog + Breakpad 是否都支持跨平台？

**可以，二者均为跨平台组合，但职责与产物格式不同，需区分理解。**

| 组件 | 跨平台情况 | 说明 |
|------|------------|------|
| **spdlog** | **支持** Windows、Linux、macOS（及常见嵌入式 Linux） | 与 **§4.6.1** 一致；同一套 API，仅 sink/路径/编码需按 OS 做少量配置。 |
| **Breakpad** | **支持** Windows、Linux、macOS（及 Android 等，若未来需要） | Google 系经典方案：各平台用 **同一套集成思路**（client 进进程、崩溃时写 **dump + metadata**），但 **各 OS 上 dump 文件格式并不相同**——Windows 侧常见为 **MiniDump**，Linux/macOS 侧为 **Breakpad 自有格式**；符号化依赖 **`dump_syms` / `minidump_stackwalk` 等工具链**，需在 CI/发布流程中 **为各平台保留调试符号** 并文档化解析步骤。 |
| **Crashpad** | **支持**多平台 | Google 在 Breakpad 之后的演进方向之一，API 与集成方式不同，同样可走 **统一目录 + 服务端符号化**；选型时对比 **维护活跃度、Qt/静态链接、体积** 与团队熟悉度。 |

**结论**：**日志用 spdlog、崩溃用 Breakpad** 作为 **跨平台技术选型没有问题**；要注意的是 **「跨平台」指同一套代码与流程可覆盖多 OS**，而不是所有平台上生成 **同一种 .dmp 文件**。若产品强依赖「全平台只有一种 dump 格式」，需在方案阶段明确（例如仅 Windows 用 minidump、Linux 用 Breakpad 格式并由统一服务器转换/解析）。

## 5. 建议优化的细节（工程化）

### 5.1 共享内存协议（核心）

- **固定布局头**：魔数、协议版本、帧序号、时间戳、数据类型、各 plane 偏移与长度、校验（CRC32 或 Adler32）。
- **多缓冲区环形队列**：至少 **双缓冲或三缓冲**，中枢写、算法读，避免长时间持锁。
- **背压**：检测/重建慢于采集时，中枢应 **丢帧策略可配置**（最新帧优先 / 阻塞 / 告警），并在 gRPC 中上报 **队列深度**。
- **对齐**：按 64 字节缓存行对齐大块区，减少伪共享（多核 CPU）。

### 5.2 gRPC API 设计

- **控制面**：`Connect` / `Configure` / `TriggerCapture` / `Abort` / `GetDeviceInfo` / `SubscribeStatus`。
- **数据面**：单次采集结果以 **引用共享内存区名 + offset + 长度 + 序列号** 返回；可选 **小图缩略图** inline 在 RPC 内便于快速预览。
- **流**：进度、日志、温度/丢包统计用 **server streaming**；避免用同一 stream 传大图。
- **兼容性**：proto 字段只增不改，**`reserved`** 废弃字段；客户端与服务端 **版本协商**。

### 5.3 安全与权限（开发者版 vs 用户版）

- **用户版**：仅开放稳定 API 与脱敏后的结果（例如仅缺陷框与简化统计）。
- **开发者版**：原始 RAW、中间特征、标定内参等；建议 **本地证书或 License 文件** 由中枢校验，避免误配在生产环境。
- **共享内存权限**：Windows 上限制 **命名对象** ACL；Linux 上注意 **`/dev/shm` 权限** 与 **运行用户**，避免多用户环境下误连他人内存区。

### 5.4 部署与运维

- **进程托管（跨平台）**：Windows 可用 **服务 / 任务计划**；Linux 可用 **systemd** user/unit；中枢拉起检测/重建子进程并监控，**PID 文件与优雅退出** 行为在两平台保持一致语义。
- **健康检查**：心跳、最后一帧时间、串口连续失败次数、相机掉线重连。
- **配置**：单一 **YAML/JSON** + schema 校验；相机与串口参数与现场标定文件版本绑定。

### 5.5 调试与一体机界面

- 调试 UI 建议 **仅绑定本机 127.0.0.1** 或 **管理口令**，避免现场网络暴露。
- 日志与崩溃策略以 **§4.6** 为准；用户版默认 **Warn**，开发者版可开 **Debug**（见公共日志模块配置）。

### 5.6 使用 Boost.Interprocess 时如何保证「实时性」（低延迟与可预期）

**先澄清术语**：在常规 **Windows / Linux 桌面或服务器内核**上，Boost.Interprocess **不提供硬实时（RTOS 级）保证**；能做到的是 **尽量低的端到端延迟** 与 **可统计、可配置的抖动上界**（软实时）。若必须硬实时，需 **RTOS、隔离 CPU、PREEMPT_RT（Linux）** 等单独方案，与本文 IPC 选型正交。

在 **Boost.Interprocess + 共享内存映射** 前提下，建议从以下几方面保证交互「够快、够稳」：

| 手段 | 做法 |
|------|------|
| **数据路径少锁或无锁** | 大块数据放在 **单段 `mapped_region`** 内，用 **SPSC/MPSC 环形缓冲** + **`std::atomic` 序列号 / 内存序**（`release`/`acquire`）协调；避免在热路径长时间持有 **跨进程 `interprocess_mutex`**。控制信令与「第 N 槽可读/可写」用原子更新即可。 |
| **零拷贝或单次拷贝** | 生产者直接写入槽位；消费者读 **已提交序号** 后的视图；仅在厂商 SDK 强制要求时再 `memcpy` 到共享区。 |
| **减少缺页与分配抖动** | 启动时 **一次性** 创建并 `map` 足够大的共享区，**预触（touch）** 或 **mlock/VirtualLock**（慎用：占物理内存、需权限）关键缓冲区，避免首帧/峰值时 **页错误** 拉长尾延迟。 |
| **通知 vs 轮询** | **低延迟**：消费者可 **短自旋 + `pause`/yield** 等待序号（微秒级，占 CPU）。**省 CPU**：用 **`interprocess_semaphore`** 或 **命名 event/条件变量类** 唤醒；唤醒有一次 **内核态切换** 成本，通常仍远小于 gRPC 传大图。按场景二选一或 **自旋限时后阻塞**。 |
| **避免伪共享** | 头结构、序号与 **大块 payload** 的缓存行隔离（见 §5.1 对齐）；多消费者时考虑 **每消费者独立序号区**。 |
| **与 RPC 解耦** | **gRPC 只传「第几槽、序号、时间戳」**；真正像素/点云只在共享内存内传递，避免把实时路径绑到网络栈。 |
| **进程与线程优先级** | Linux：采集/编解码线程可考虑 **`SCHED_FIFO`** 与 **CPU 亲和**（需谨慎，防止饿死系统）。Windows：适当提高进程/线程优先级，并避免与 **高优先级 UI / 杀毒实时扫描** 争用同一核。 |
| **背压显式化** | 槽满时 **丢弃最旧帧或阻塞写端**（策略可配），避免无限堆积导致 **延迟爆炸**；监控 **队列深度与帧龄（age）** 比单纯「吞吐」更能反映实时性。 |

**小结**：实时性主要靠 **无锁环缓冲 + 预分配共享区 + 合理的唤醒策略 + 与 gRPC 分离的大数据路径** 来保证；Boost.Interprocess 提供的是 **跨进程映射与同步原语**，本身不替代上述数据面设计。若需把量化指标写进合同，建议约定 **P99 帧龄 / 端到端延迟** 与测试场景（分辨率、并发进程数）。

### 5.7 公共日志与崩溃模块的工程约定（落地检查清单）

- **单一入口**：每个 `main` 仅调用 **`PlatformDiag::Init(config)`**（命名示例）一次；配置来自 **统一配置文件** 或环境变量（日志目录、级别、异步队列长度、是否启用 minidump）。
- **禁止分叉实现**：业务代码不直接 `std::cout` / 原始 `fopen` 写诊断文件；第三方库噪声可通过 **回调适配** 或 **sink 过滤** 收敛。
- **栈回溯**：Debug 构建可开 **完整符号**；Release 保留 **PDB（Windows）/ 分离 debuginfo（Linux）** 供事后解析 minidump / addr2line。
- **测试**：CI 中增加 **故意崩溃用例**（子进程）验证 **dump 生成与日志尾部一致性**；避免 handler 内再抛异常导致死循环。
- **一键导出**：与 **§6** 中「现场日志包」对齐：脚本或菜单项打包 **各进程最近日志 + 最近一次 crash 文件 + 版本清单**。

## 6. 迭代路线建议（简）

1. **MVP**：单进程相机采集 + 假检测（直通）+ gRPC 返回路径验证；共享内存双缓冲协议冻结 v0；**公共日志库 + 崩溃捕获在首个可执行文件中落地**，后续进程只复用不重写。
2. **V1**：检测、重建独立进程；单次采集状态机；Viewer 深度图 + 结果列表。
3. **V2**：开发者/用户双包；性能剖析与零拷贝优化；看门狗与现场日志包一键导出。
