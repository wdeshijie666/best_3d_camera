# 内存与线程安全（调用链审查）

## 何时执行

- **新增需求**或**修改需求**且本轮将修改 **`.cpp` / `.h`** 或其它**影响运行时的逻辑**（不仅改注释、不改行为时可弱化，但仍建议快速过一遍清单）。
- 在**开始写具体实现之前**先读本技能；**提交改动前**再沿**整条调用链**从入口到出口做一次核对。

## 调用链范围

从用户故事涉及的**对外入口**起追踪到数据落点，例如：

- gRPC 方法 → Hub 内部状态 → 相机/串口/编排 → SHM 写入/读出；
- SDK 公开 API → 通道与 stub → 回调；
- 跨线程：采集回调、定时器、`std::thread` / 线程池、异步 `future`。

**整条链**上的中间对象、缓存、回调捕获、析构顺序都要纳入审查，而不是只改动的单个函数。

---

## 内存泄漏与资源生命周期

实现时**主动避免**以下问题；审查时逐项对照：

| 风险 | 注意点 |
|------|--------|
| **所有权不清** | `new`/`delete` 不成对；应优先 `unique_ptr`/`shared_ptr` 或栈对象；接口边界写清「谁分配谁释放」。 |
| **循环 `shared_ptr`** | 回调或子对象反向持有父对象导致环；必要时 `weak_ptr` 或明确断开回调。 |
| **异常路径** | `try` 中分配的资源在 `catch` 外泄漏；RAII 封装或 `unique_ptr` 接管。 |
| **容器与缓存** | 无界 `vector`/map 增长；队列不 pop；环形缓冲槽位「只写不回收」语义是否与文档一致。 |
| **SHM / 映射区** | `mmap`/Boost.Interprocess 区段生命周期与进程退出顺序；是否重复 `remove`/`unlink` 或泄漏映射。 |
| **C API / 第三方** | `*_create` / `*_destroy` 成对；grpc/回调里持有没有释放的句柄。 |
| **lambda 捕获** | 按引用捕获已销毁对象；延长生命周期是否用 `shared_from_this` 等需一致且可析构。 |

## 线程安全

实现时**主动避免**数据竞争与死锁：

| 风险 | 注意点 |
|------|--------|
| **共享可变状态** | 多线程读写的成员、静态全局、单例缓存；若无同步则加 `mutex`、用 `atomic`、或线程局部/只读初始化。 |
| **先检查后使用** | 「if (!ptr)」与使用之间被其它线程改掉；用局部拷贝、锁保护或原子指针。 |
| **回调线程** | gRPC `CompletionQueue`/异步回调、相机 SDK 回调在哪个线程执行；与 Hub 主状态交互时是否都在同一把锁或队列上串行化。 |
| **锁顺序** | 多把锁固定全局顺序，避免 AB-BA 死锁；锁内避免调用可能再次加锁的用户代码。 |
| **条件变量** | 用 `while` 防止虚假唤醒；修改条件与 `notify` 的次序正确。 |
| **const 正确性** | 在锁保护下读写的数据避免从 `const` 方法无锁修改（常见疏漏）。 |

---

## 落地顺序（建议）

1. **画清或列出**本需求涉及的调用链与线程边界（一两句即可）。
2. **按上表**对链上每个节点做泄漏与并发标注；发现缺口则在实现中补上（RAII、锁、`weak_ptr`、无锁队列等）。
3. **实现**时优先简单可证明正确的同步，避免过度「无锁优化」除非有测量依据。
4. **改完后**再快速扫一遍：新增路径是否在异常/早退时释放资源；是否在锁内做了可能阻塞的 IO。
