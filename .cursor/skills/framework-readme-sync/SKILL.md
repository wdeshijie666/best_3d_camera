# 框架变更与 README.md 同步

## 何时必须先读并执行本技能

- 用户发来**需求、任务、改动说明**（含「加功能」「重构」「联调」「新服务/新库」等），且准备动代码或 CMake 之前。
- 本轮会改到下列任一类：**根 `CMakeLists.txt`**、**`cmake/`**、**`proto/`** 对外契约、**`services/`**、**`libs/`**、**`sdk/`**、**`tools/`** 下**新增/删除/重命名**顶层子项或显著改变模块职责、**与 README 中 mermaid/表格已描述路径不一致**的改动。

## 判定：是否「框架相关」

满足**任一**即视为框架相关，**必须在完成实现后更新 `README.md`**：

1. **拓扑变化**：增删改 `services/*`、`libs/*`、`sdk/*`、`tools/*` 的**顶层目录**或其在根 CMake 中的「一级」接入方式。
2. **契约/控制面**：`proto/` 中影响「谁调谁、传什么」的变更；Hub/SDK 间状态码或会话模型在**文档层**需对齐时（可指向已有小节或 `README_hub_recon_grpc_shm.md`）。
3. **数据面总览**：共享内存区名、环形缓冲角色、Hub↔Recon 数据路径在架构意义上改变，且总览 `README.md` 应反映（细节仍可引用 `README_hub_recon_grpc_shm.md`）。
4. **构建与开关**：根 CMake **选项表**（当前对应 README **§5**）含义变化，或默认/推荐构建组合改变。
5. **用户显式**要求更新总览、架构图、目录说明。

**通常不算框架**（无需改 README，除非用户点名）：单文件 bugfix、日志文案、某库内部实现无对外可见边界变化、仅注释、与 README 已写结构完全一致的局部逻辑。

不确定时：**宁可更新 README 中一句「说明」或表格脚注**，也不要留过期总览。

## 执行顺序

1. **先判定**：用上文清单打勾；若框架相关，在本轮待办中记下「同步 README」。
2. **改代码/CMake**：按用户要求完成实现。
3. **再读 README**：用 Read 读取仓库根目录 **`README.md`**（路径：`README.md`，与 `d:\best_project\README.md` 同文件）。
4. **同步内容**（保持现有风格：中文、**UTF-8**、标题层级、mermaid、Markdown 表格）：
   - **§1 逻辑架构**：流程图节点或连线与真实调用关系一致；新增/删除进程或库层大块时更新 `flowchart`。
   - **§2 目录与构建产物**：表格增删行或改「角色 / 技术要点」；新库新服务必须有一行（路径用反引号包一层目录名，与仓库一致）。
   - **§3 典型运行流程**：启动参数、环境变量、会话/捕获/SHM 步骤与代码一致。
   - **§4 交互技术要点**：易错点与当前实现一致。
   - **§5 常用 CMake 选项**：选项名与含义与根 `CMakeLists.txt` / `cmake/` 一致。
   - **§6 代码注释约定**：仅当注释规范或引用文件（如 `code-comments-skill.txt`）变更时改。
5. **自检**：保存后心中过一遍「新同事只看 README 能否理解当前顶层结构」；**不要**为凑字数大段重写无关章节。

## 路径与约束

- 唯一总览文件：**仓库根 `README.md`**（Windows 工作区绝对路径为 `D:\best_project\README.md` 时仍为同一文件）。
- 与 Hub/Recon/SHM **细时序**强相关的内容优先放在 **`README_hub_recon_grpc_shm.md`**，README 保持**简引**+链接，避免两处长篇重复；但总览中的**模块边界与数据走向**仍须在 README 与代码一致。
- 不引入与用户改动无关的「顺手重构」式文档改写。

## 小结

**每轮需求** → 先判断是否框架相关 → 若是，**同一轮内**在改完代码后 **更新 `README.md` 对应小节**，使总览持续为真。
