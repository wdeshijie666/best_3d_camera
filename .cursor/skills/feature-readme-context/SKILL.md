# 功能类需求与 README 上下文

## 何时执行本技能

- 用户描述涉及：**新增功能**、**功能修改**、**行为/接口变更**、**联调方式变化**，且将进入代码实现或排查时。

## 第一步：判断与「整仓架构总览」的关联度

在读取 `README.md` 之前，用一两句内心判断（不必输出长文）归类为 **高 / 中 / 低**：

| 关联度 | 典型情况 |
|--------|----------|
| **高** | 跨 `services` / `libs` / `sdk` / `proto` 边界；动到 Hub、SHM、gRPC、会话/捕获主流程；新增或调整进程、库在总览中的角色。 |
| **中** | 单模块内较大改动，但可能依赖 README 中的**控制面/数据面**约定、CMake 选项、典型启动流程；或不确定是否影响对外契约。 |
| **低** | 局部 bugfix、命名/日志、单文件内小范围逻辑、与 README 已述拓扑明显无关的纯实现细节。 |

**原则**：不强求每个需求都「套进」总览架构；**低关联时不要为了形式去硬关联架构叙事**。

## 第二步：是否阅读 `README.md`

- **高或中**：在**开始改代码或做较大重构之前**，用 Read 读取仓库根目录 **`README.md`**（与工作区根下 `README.md` 相同；绝对路径 `D:\best_project\README.md` 指向同一文件）。关注：逻辑架构图、目录表、典型运行流程、交互要点、CMake 表等与当前任务可能相交的小节即可，不必每次背诵全文。
- **低**：**不强制**整篇阅读 `README.md`；以用户描述 + 相关源码为准。若实现过程中发现可能触及总览中的模块边界或 SHM/gRPC 约定，再**补读** `README.md` 对应段落（或 `README_hub_recon_grpc_shm.md`）。

## 与其它技能的关系

- **`framework-readme-sync`**：在判定为框架相关并改完代码后，负责**回写** `README.md`；本技能负责在**动手前**按需**读取** `README.md` 以理解上下文，二者互补。
- 细时序、Hub↔Recon 专题仍以 **`README_hub_recon_grpc_shm.md`** 为主；总览不清时再深入。

## 小结

**功能新增/修改** → **先判与总览架构的关联度** → **高/中则先读 README.md** → **低则不勉强，按需补读**。
