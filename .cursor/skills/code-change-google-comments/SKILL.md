# 代码改动时的 Google 风格配套注释

## 前置（不得省略）

1. 用 Read 读取 **`.cursor/skills/chinese-code-comments/SKILL.md`**，所有「精简、中文、禁止废话」等准则**以该技能为准**，本文件只规定**配套范围与分层**。
2. 若涉及仓库注释总原则，可对照根目录 **`code-comments-skill.txt`**（与 `README.md` §6 一致）。

## 何时必须执行

- 本轮将**新增或修改**实现代码（至少包括 **`.cpp` / `.h`**，以及团队视为「源码」的同类文件）时：**同一轮提交中**注释与代码一起交付，禁止「先裸提交、后补注释」除非用户明确只要草稿。

## 三层注释（Google 习惯 + 中文）

### 1. 文件头：模块作用

- 在文件**最前**（在 `#include` 之前或紧跟许可块之后，与项目现有风格一致）用块注释写清：
  - **本文件职责**（一句）；必要时一句 **关键依赖**（如「Hub gRPC 服务实现」「共享内存槽位写入」）。
- 可用 **`@file`** 行或等价标题，保持**一行到数行**，不复制 README 全文。

### 2. 接口注释（对外 API）

- **头文件**中的 `class` / `public` 方法 / 重要 `struct` / 回调类型：在声明**上方**用块注释说明：
  - **做什么**、**关键前置条件**（线程、会话、已 Connect 等）、**错误/返回值要点**（可与业务码文档交叉引用，不贴长篇）。
- 遵循 **Google**：声明处写契约与注意点；**不在头里写实现细节**。
- **私有成员**：仅当名称不足以表达不变量或跨线程约束时再加简短注释。

### 3. 实现文件内的代码行注释

- 在 **`.cpp`** 中，对**非显然**分支、单位、时序、`seq`/代际、锁顺序、与 proto 字段对应关系等，用 **`//` 行尾或紧邻上一行** 简短说明。
- **显然**的代码（`i++`、getter 单行返回）不写行注释。
- 较长函数：可在**逻辑段**前用一行分段注释（仍用中文，忌散文）。

## 与「已有代码」一致

- 同一目录或相邻文件已有注释格式（`//` 块、`/* */`）时，**优先对齐现有文件**，不引入另一种排版风格。

## 改代码时的同步义务

- **改签名 / 改语义**：更新接口块注释与相关行注释；删除过时注释。
- **删代码**：删除已无对应实现的注释。

## 自检（提交前扫一眼）

- 新文件是否有**文件头**？对外新符号在 **`.h`** 是否有**接口注释**？
- `.cpp` 里复杂路径是否有点睛注释？是否违反 `chinese-code-comments` 中的禁止项？
